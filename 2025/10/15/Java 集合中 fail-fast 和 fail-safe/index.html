Java 集合中 fail-fast 和 fail-safe

2025-10-15 · 2 min · 570 words · Java 集合框架

👏——这是 Java 集合框架中一个经常考的核心概念，尤其是和并发修改异常（ConcurrentModificationException）相关。
我们来分层次地讲清楚：

🧩 一、基本概念

✅ Fail-Fast（快速失败）

当多个线程或同一线程在遍历集合的过程中，集合结构被修改（非通过迭代器自身的 remove() 方法），会立刻抛出异常，而不是继续遍历。

  • 代表集合：

    • ArrayList
    • HashMap
    • HashSet
    • LinkedList
      等 非并发集合。

  • 原因：

    • Iterator 内部维护了一个 expectedModCount（预期修改次数）
    • 集合本身维护了 modCount（实际修改次数）
    • 每次迭代比较两者，不相等就抛异常。

✅ Fail-Safe（安全失败）

在遍历过程中，即使集合被修改，也不会抛异常。因为它遍历的是原集合的副本（快照）。

  • 代表集合：

    • CopyOnWriteArrayList
    • CopyOnWriteArraySet
    • ConcurrentHashMap
    • ConcurrentSkipListMap 等 并发集合类。

  • 特征：

    • 修改集合不会影响当前遍历结果（因为遍历的快照）
    • 但修改后的新元素，不会被本次遍历看到

🧠 二、对比总结表

特性 Fail-Fast Fail-Safe
出现位置 非并发集合 并发集合
修改检测机制 modCount 与 expectedModCount 比较 通过快照机制或并发控制
是否抛异常 是 (ConcurrentModificationException) 否
遍历的数据 原集合 集合副本（快照）
实现类举例 ArrayList, HashMap, HashSet ConcurrentHashMap, CopyOnWriteArrayList
适用场景 单线程安全修改 多线程并发读写

⚙️ 三、延伸理解

  • Fail-Fast 并不是绝对可靠的机制：
    它是尽最大努力地抛出异常，并非保证检测所有并发修改。

  • Fail-Safe 的代价：

    • 内存消耗大（需要复制数据）
    • 性能相对较低（尤其在频繁写操作时）

🧩 四、小结口诀

“快失败，抛异常；
安全型，做快照。
非并发，用 fail-fast；
并发用 fail-safe。”
